# Performance report

See [README.md](README.md) for the method.

## Findings

Input: `germany-latest.osm.pbf` (4.4 GB). Sub-region: ~Hessen bbox. Tags: admin
boundaries + a few playground POIs (sparse). Numbers below are from the table.

**1. RAM is the headline — and it's bounded, not scary.** Every `osmium extract`
peaked around **3.7–4.6 GB**, regardless of how small the output was, because the
cost is the global node-ID index, not the data kept. So:
- osmium does **not** need RAM proportional to the 87 GB planet — but **~4 GB on a
  country file is a real floor**, and the planet will be somewhat higher (more
  ways/relations to track). **Budget ≥ 8 GB headroom for the pipeline on top of
  uMap, and re-measure on the planet** (PLAN §3 RAM is still unconfirmed).
- `tags-filter` **with** referenced objects (our default) peaked ~2.2 GB; **with
  `-R`** (omit) only **0.25 GB**.

**2. Strategy: `smart` costs little over `complete_ways`, `simple` is incomplete.**
simple 25.6 s / 2.3 GB → complete_ways 51 s / 3.7 GB → smart 63 s / 3.9 GB. smart
adds only ~12 s and ~0.2 GB over complete_ways for the same ~621 MB output, and it
completes multipolygon **relations** (boundaries) — so **`smart` is the right
default for boundary projects**; `complete_ways` is a fine, slightly cheaper choice
where relations aren't needed. (This is exactly what the per-project
`extract_strategy` knob now selects.)

**3. `-R` is a 3×-time / ~9×-RAM lever — but it drops geometry.** tags-filter
include = 66 s / 2.2 GB / 53 MB; omit (`-R`) = 23.5 s / 0.25 GB / 16 MB. The cheap
mode produces a much smaller file precisely because it **drops referenced
nodes/members**, leaving ways/relations without geometry. We want the **default
(include)** for usable extracts — which validates the `-R` correctness fix
(include-by-default; `-R` only when a project opts out).

**4. Order: for sparse tags, tag-then-region is marginally faster.** tag→region
74.7 s vs region→tag 82.4 s. Tag-filtering the country first shrinks it to 53 MB,
making the region cut almost free (3.1 s vs 9.6 s). The dominant cost in both is
the **one unavoidable full pass** over the input. Implication for the planet: a
**global tag-filter pass before region cuts** is worth prototyping for our sparse
tag set — but note tags-filter-with-referenced is itself a ~3-pass operation, so
on 87 GB the win is not guaranteed; **benchmark on the planet before changing the
pipeline** (it currently does region-first per osmium's own guidance).

**Bottom line for the shared server:** plan for **~4 GB+ RAM** and a handful of
**full-file passes** per run (disk-I/O bound). Run osmium under `nice`/`ionice`,
off-peak, and watch RAM/I/O in the FOSSGIS monitoring (PLAN §A8/§B8). Nothing here
suggests the work won't fit a typical server — but confirm RAM and re-run on the
planet.

> Caveats: container osmium is **1.15.0** (Debian bookworm), the server/host run
> 1.19; Docker-on-macOS wall-times are indicative (VM I/O), peak RAM transfers.
> `_env.txt` shows `time (GNU Time) UNKNOWN` — a `--version` quirk; `%M`/`%e` work,
> and per-step `%M` agrees with the cgroup `memory.peak` cross-check.
